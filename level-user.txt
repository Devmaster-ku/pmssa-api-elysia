ระบบบริหารโครงการ เพื่อขับเคลื่อนยุทธศาสตร์ มหาวิทยาลัยเกษตรศาสตร์
Project Management System for Strategic Advancement of Kasetsart University
Version: 1.0
เป็นระบบการจัดการและขออนุมัติโครงการที่ทันสมัย ใช้งานง่าย และมีประสิทธิภาพสูง สำหรับมหาวิทยาลัยเกษตรศาสตร์
ระบบเดิมรองรับผู้ใช้งานระดับหน่วยงานเท่านั้น เช่น คณะฯ โดยแบ่งสิทธิ์ผู้ใช้งานออกเป็น
- super admin
- manager
- admin
- staff
ซึ่งระบบเดิมกำหนดให้ user 1 user/account มีได้หลาย roles แต่สามารถสังกัดหน่วยงานได้เพียงสังกัดเดียวเท่านั้น
ดังนั้นจึงเกิดปัญหา คือ ต้องการให้ผู้ใช้งาน 1 user/account สามารถอยู่ได้หลายสังกัดหน่วยงาน เช่น
- นาย A อยู่ภาควิชา B1 จะสามารถอยู่สังกัด AB1 ได้ด้วย 
และระบบใหม่ต้องการให้รองรับผู้ใช้งานในระดับที่สูงขึ้น
เช่น 
 - ผู้ใช้งาน สูงสุดระดับมหาวิทยาลัย (กลุ่มบริหาร) เช่น อธิการบดี รองอธิการบดี ผู้ช่วยอธิการบดี สังกัดบางเขน สามารถดูข้อมูลได้ทุกวิทยาเขต ไม่สามารถแก้ไขข้อมูลได้
 - ผู้ใช้งาน สูงสุดระดับวิทยาเขต (กลุ่มบริหาร) รองอธิการบดี ที่สังกัดวิทยาเขตอื่น สามารถดูข้อมูลได้เฉพาะที่สังกัดวิทยาเขตตนเองเท่านั้น ไม่สามารถแก้ไขข้อมูลได้
 - ผู้ใช้งาน สูงสุดระดับมหาวิทยาลัย (กลุ่มเจ้าหน้าที่) สามารถดูข้อมูลได้ทุกวิทยาเขต เช่น กองยุทธศาสตร์ ไม่สามารถแก้ไขข้อมูลได้
 - ผู้ใช้งาน สูงสุดระดับวิทยาเขต (กลุ่มเจ้าหน้าที่) สามารถดูข้อมูลได้เฉพาะที่สังกัดวิทยาเขตตนเองเท่านั้น เช่น สำนักงานวิทยาเขต ไม่สามารถแก้ไขข้อมูลได้
 - ผู้ใช้งาน สูงสุดระดับหน่วยงาน (กลุ่มบริหาร) เช่น ผู้อำนวยการฯ/คณบดี สามารถดูข้อมูลได้ทุกภาควิชาที่สังกัดหน่วยงานหลักเท่านั้น เช่น คณะฯ ไม่สามารถแก้ไขข้อมูลได้
 - ผู้ใช้งาน สูงสุดระดับหน่วยงาน (กลุ่มบริหารและเจ้าหน้าที่) เช่น หัวหน้าสำนักงาน/หัวหน้าภาควิชา/หัวหน้าสถานีวิจัย สามารถดูข้อมูลได้เฉพาะที่สังกัดเท่านั้น เช่น สำนักงานเลขานุการ/ภาควิชา เป็นต้น
 - ผู้ใช้งาน ระดับหน่วยงาน (ผู้ดูแลระบบระดับหน่วยงาน) เช่น admin สามารถดูข้อมูลได้เฉพาะที่สังกัดเท่านั้น เช่น คณะฯ สถานีวิจัย เป็นต้น สามารถแก้ไขข้อมูลได้ และกำหนดสิทธิ์ผู้ใช้งานระดับหน่วยงานหลักและหน่วยงานย่อยได้
 - ผู้ใช้งาน ระดับหน่วยงาน (กลุ่มผู้รับผิดชอบโครงการ) เช่น หัวหน้าโครงการ สามารถดูข้อมูลหรือแก้ไขข้อมูลได้เฉพาะที่ตนเองเป็นหัวหน้าโครงการเท่านั้น
 - ผู้ใช้งาน ระดับหน่วยงาน (กลุ่มอาจารย์/เจ้าหน้าที่) เช่น ผู้ที่สามารถเข้าถึงโครงการสามารถจัดการข้อมูลและเอกสารโครงการได้ สามารถดูข้อมูลได้เฉพาะที่สังกัดเท่านั้น เช่น สำนักงานเลขานุการ/ภาควิชา เป็นต้น

ข้อมูลเพิ่มเติม
บางเขน คือ มหาวิทยาลัยเกษตรศาสตร์ กลาง ไม่นับเป็นวิทยาเขต แต่ส่วนอื่นยังคงเดิม
และ แนวคิดการกำหนดสิทธิ์ในระบบ 
เมื่อ ผู้ใช้งานสามารถสังกัดได้หลายหน่วยงาน ก็จะมีหลายตำแหน่งด้วย เช่น 
User: นาย A เป็นหัวหน้าภาควิชา AA และมีตำแหน่งเป็นประธานโครงการหลักสูตร ของ โครงการปริญญาตรีภาคพิเศษด้วย หรือ 
User: นาย B เป็นอาจารย์ประจำภาควิชาวิทยาการคอมพิวเตอร์ คณะวิศวกรรมศาสตร์ ศรีราชา และมีตำแหน่งเป็นรองอธิการบดีระดับวิทยาเขต มีตำแหน่งที่สำนักงานวิทยาเขต

1. บทนำ (Introduction)
    1.1 วัตถุประสงค์ของเอกสาร
    เอกสารนี้เป็นข้อกำหนดความต้องการระบบ (System Requirements Specification) สำหรับระบบบริหารโครงการเพื่อขับเคลื่อนยุทธศาสตร์ มหาวิทยาลัยเกษตรศาสตร์ เวอร์ชันใหม่ ซึ่งออกแบบมาเพื่อแก้ไขข้อจำกัดของระบบเดิม และรองรับโครงสร้างการบริหารงานที่ซับซ้อนของมหาวิทยาลัย
    1.2 ปัญหาของระบบเดิม
    ระบบเดิมมีข้อจำกัดสำคัญดังนี้:
        - รองรับผู้ใช้งานระดับหน่วยงาน (คณะ/สำนัก) เท่านั้น ไม่ครอบคลุมระดับมหาวิทยาลัยและวิทยาเขต
        - ผู้ใช้งาน 1 user/account สามารถสังกัดได้เพียงหน่วยงานเดียว ทำให้ไม่สอดคล้องกับความเป็นจริงที่บุคลากรอาจมีบทบาทในหลายหน่วยงาน
        - ไม่รองรับการมองเห็นข้อมูลข้ามหน่วยงานสำหรับผู้บริหารระดับสูง

    1.3 เป้าหมายของระบบใหม่
        - รองรับผู้ใช้งาน 1 user/account สังกัดได้หลายหน่วยงาน พร้อมตำแหน่งที่แตกต่างกันในแต่ละสังกัด
        - รองรับโครงสร้างการบริหารงาน 3 ระดับ ได้แก่ มหาวิทยาลัย วิทยาเขต และหน่วยงาน
        - กำหนดสิทธิ์การเข้าถึงข้อมูลอย่างละเอียด (Granular Access Control) ตามบทบาทและระดับสังกัด
        - รองรับทั้งสิทธิ์ดูข้อมูล (View-only) และสิทธิ์แก้ไขข้อมูล (Edit) ตามบทบาท

2. โครงสร้างองค์กร (Organizational Structure)
    2.1 ลำดับชั้นองค์กร (Organization Hierarchy)

        มหาวิทยาลัยเกษตรศาสตร์ (ส่วนกลาง - บางเขน)
        ├── วิทยาเขตกำแพงแสน
        │   ├── คณะ / สำนัก / สถาบัน (หน่วยงานหลัก)
        │   │   ├── สำนักงานเลขานุการ (หน่วยงานย่อย)
        │   │   ├── ภาควิชา (หน่วยงานย่อย)
        │   │   ├── สถานีวิจัย (หน่วยงานย่อย)
        │   │   └── โครงการพิเศษ (หน่วยงานย่อย)
        │   └── สำนักงานวิทยาเขต
        ├── วิทยาเขตศรีราชา
        │   ├── คณะ / สำนัก / สถาบัน (หน่วยงานหลัก)
        │   │   └── ... (หน่วยงานย่อย)
        │   └── สำนักงานวิทยาเขต
        ├── วิทยาเขตเฉลิมพระเกียรติ จ.สกลนคร
        │   ├── คณะ / สำนัก / สถาบัน (หน่วยงานหลัก)
        │   │   └── ... (หน่วยงานย่อย)
        │   └── สำนักงานวิทยาเขต
        ├── คณะ / สำนัก / สถาบัน ส่วนกลาง (หน่วยงานหลัก)
        │   └── ... (หน่วยงานย่อย)
        └── หน่วยงานสนับสนุนส่วนกลาง
            ├── กองยุทธศาสตร์
            └── สำนักงานมหาวิทยาลัย

    2.2 หมายเหตุสำคัญ
        - บางเขน คือส่วนกลางของมหาวิทยาลัย ไม่นับเป็นวิทยาเขต ผู้บริหารที่สังกัดบางเขนถือเป็น ระดับมหาวิทยาลัย
        - วิทยาเขต ได้แก่ กำแพงแสน ศรีราชา และเฉลิมพระเกียรติ จ.สกลนคร
        - หน่วยงานหลัก หมายถึง คณะ สำนัก สถาบัน หรือหน่วยงานเทียบเท่า
        - หน่วยงานย่อย หมายถึง ภาควิชา สำนักงานเลขานุการ สถานีวิจัย โครงการพิเศษ ฯลฯ

    2.3 ตาราง Entity: Organization
        Field               Type                Description
        org_id              UUID                รหัสหน่วยงาน (Primary Key)
        org_name_th         VARCHAR             ชื่อหน่วยงาน (ภาษาไทย)
        org_name_en         VARCHAR             ชื่อหน่วยงาน (ภาษาอังกฤษ)
        org_level           ENUM                ระดับหน่วยงาน: university, campus, faculty, department
        parent_org_id       UUID (FK)           รหัสหน่วยงานแม่ (NULL สำหรับมหาวิทยาลัย)
        campus_id           UUID (FK)           รหัสวิทยาเขตที่สังกัด (NULL สำหรับส่วนกลาง)
        is_active           BOOLEAN             สถานะการใช้งาน

    3.3 หลักการคำนวณสิทธิ์ (Permission Resolution)
    เมื่อผู้ใช้มีหลายสังกัดและหลายบทบาท ระบบจะ:
        1. ใช้หลัก Union of Permissions — สิทธิ์ที่ผู้ใช้ได้รับคือผลรวมของสิทธิ์จากทุกสังกัด
        2. แยก Context ตามสังกัด — ผู้ใช้เลือก Context (สังกัด) ที่ต้องการทำงานผ่าน Session Switcher
        3. View-only ไม่ถูก Override — หากบทบาทหนึ่งกำหนดให้ดูอย่างเดียว สิทธิ์แก้ไขจากบทบาทอื่นจะไม่ส่งผลข้ามสังกัด

    project_lead         หัวหน้าโครงการ                       หน่วยงาน                        ผู้รับผิดชอบโครงการ       เฉพาะโครงการที่ตนเองรับผิดชอบ                           ดู + แก้ไข (เฉพาะโครงการ)
    staff                อาจารย์/เจ้าหน้าที่                     หน่วยงาน                        อาจารย์/เจ้าหน้าที่        เฉพาะหน่วยงานย่อยที่สังกัด                                ดู + แก้ไข (เฉพาะที่ได้รับสิทธิ์)

    4.2 รายละเอียดแต่ละบทบาท
        4.2.1 super_admin — ผู้ดูแลระบบสูงสุด
        - ขอบเขต: ทุกข้อมูลในระบบทั้งหมด
        - สิทธิ์: CRUD ทุกข้อมูล + จัดการผู้ใช้งาน + กำหนดค่าระบบ
        - ตัวอย่าง: ทีมพัฒนาระบบ / ผู้ดูแลระบบสารสนเทศมหาวิทยาลัย

        4.2.2 univ_executive — ผู้บริหารระดับมหาวิทยาลัย (กลุ่มบริหาร)
        - ขอบเขต: มองเห็นข้อมูลทุกวิทยาเขต ทุกหน่วยงาน
        - สิทธิ์: ดูอย่างเดียว (Read-only)
        - ตัวอย่าง: อธิการบดี, รองอธิการบดี (สังกัดบางเขน), ผู้ช่วยอธิการบดี (สังกัดบางเขน)
        - หมายเหตุ: สังกัดบางเขน = ระดับมหาวิทยาลัย (ส่วนกลาง)

        4.2.3 univ_officer — เจ้าหน้าที่ระดับมหาวิทยาลัย (กลุ่มเจ้าหน้าที่)
        - ขอบเขต: มองเห็นข้อมูลทุกวิทยาเขต ทุกหน่วยงาน
        - สิทธิ์: ดูอย่างเดียว (Read-only)
        - ตัวอย่าง: เจ้าหน้าที่กองยุทธศาสตร์

        4.2.4 campus_executive — ผู้บริหารระดับวิทยาเขต (กลุ่มบริหาร)
        - ขอบเขต: มองเห็นข้อมูลเฉพาะวิทยาเขตที่ตนเองสังกัด
        - สิทธิ์: ดูอย่างเดียว (Read-only)
        - ตัวอย่าง: รองอธิการบดีวิทยาเขตศรีราชา, รองอธิการบดีวิทยาเขตกำแพงแสน

        4.2.5 campus_officer — เจ้าหน้าที่ระดับวิทยาเขต (กลุ่มเจ้าหน้าที่)
        - ขอบเขต: มองเห็นข้อมูลเฉพาะวิทยาเขตที่ตนเองสังกัด
        - สิทธิ์: ดูอย่างเดียว (Read-only)
        - ตัวอย่าง: เจ้าหน้าที่สำนักงานวิทยาเขต

        4.2.6 faculty_executive — ผู้บริหารระดับหน่วยงานหลัก (กลุ่มบริหาร)
        - ขอบเขต: มองเห็นข้อมูลทุกหน่วยงานย่อยภายใต้หน่วยงานหลักที่สังกัด
        - สิทธิ์: ดูอย่างเดียว (Read-only)
        - ตัวอย่าง: คณบดี, ผู้อำนวยการสำนัก/สถาบัน

        4.2.7 unit_head — หัวหน้าหน่วยงานย่อย (กลุ่มบริหารและเจ้าหน้าที่)
        - ขอบเขต: มองเห็นข้อมูลเฉพาะหน่วยงานย่อยที่สังกัด
        - สิทธิ์: ดูอย่างเดียว (Read-only)
        - ตัวอย่าง: หัวหน้าสำนักงานเลขานุการ, หัวหน้าภาควิชา, หัวหน้าสถานีวิจัย

        4.2.8 org_admin — ผู้ดูแลระบบระดับหน่วยงาน
        - ขอบเขต: เฉพาะหน่วยงานที่สังกัด (ทั้งหน่วยงานหลักและหน่วยงานย่อยภายใต้)
        - สิทธิ์: ดู + แก้ไข + กำหนดสิทธิ์ผู้ใช้งานระดับหน่วยงานหลักและหน่วยงานย่อย
        - ตัวอย่าง: Admin ประจำคณะ, Admin ประจำสถานีวิจัย

        4.2.9 project_lead — หัวหน้าโครงการ
        - ขอบเขต: เฉพาะโครงการที่ตนเองเป็นหัวหน้าโครงการ
        - สิทธิ์: ดู + แก้ไข (เฉพาะโครงการของตนเอง)
        - ตัวอย่าง: หัวหน้าโครงการวิจัย, หัวหน้าโครงการบริการวิชาการ

        4.2.10 staff — อาจารย์/เจ้าหน้าที่
        - ขอบเขต: เฉพาะหน่วยงานย่อยที่สังกัด + โครงการที่ได้รับสิทธิ์เข้าถึง
        - สิทธิ์: ดู + แก้ไข (เฉพาะข้อมูลและเอกสารโครงการที่ได้รับสิทธิ์)
        - ตัวอย่าง: อาจารย์ประจำภาควิชา, เจ้าหน้าที่ประจำสำนักงานเลขานุการ

    6.2 Scope Resolution Rules
        getAccessibleOrgs(user, affiliation):

            CASE role OF:

                'super_admin':
                RETURN all_organizations

                'univ_executive', 'univ_officer':
                RETURN all_organizations (READ-ONLY)

                'campus_executive', 'campus_officer':
                campus = affiliation.org.campus_id
                RETURN organizations WHERE campus_id = campus (READ-ONLY)

                'faculty_executive':
                faculty = affiliation.org_id
                RETURN organizations WHERE org_id = faculty
                        OR parent_org_id = faculty (READ-ONLY)

                'unit_head':
                RETURN organizations WHERE org_id = affiliation.org_id (READ-ONLY)

                'org_admin':
                org = affiliation.org_id
                RETURN organizations WHERE org_id = org
                        OR parent_org_id = org (READ + WRITE)

                'project_lead':
                RETURN projects WHERE lead_user_id = user.user_id (READ + WRITE)

                'staff':
                RETURN organizations WHERE org_id = affiliation.org_id (READ)
                        + projects WHERE user has access (READ + WRITE on assigned projects)

7. Session Switcher (ตัวสลับสังกัด)
    7.1 แนวคิด (Concept)
    เนื่องจากผู้ใช้ 1 คนอาจมีหลายสังกัดและหลายบทบาท ระบบจำเป็นต้องมี Session Switcher เพื่อให้ผู้ใช้เลือก Context ที่ต้องการทำงาน

    7.2 พฤติกรรม (Behavior)
        1. เมื่อ Login: ระบบแสดงรายการสังกัดทั้งหมดของผู้ใช้ หากมีสังกัดเดียว ระบบเข้าสู่สังกัดนั้นโดยอัตโนมัติ
        2. ระหว่างใช้งาน: ผู้ใช้สามารถสลับสังกัดผ่าน Dropdown/Selector ที่แสดงอยู่บน Navigation Bar
        3. การแสดงผล: แสดงชื่อหน่วยงาน + ตำแหน่ง + บทบาท ของ Context ปัจจุบัน    

    7.3 ตัวอย่าง UI Concept
        ┌──────────────────────────────────────────────────────┐
        │  🔄 สลับสังกัด                                       │
        │                                                      │
        │  ● ภาควิชาวิทยาการคอมพิวเตอร์ - อาจารย์     [active]│
        │  ○ สำนักงานวิทยาเขตศรีราชา - รองอธิการบดี           │
        │                                                      │
        └──────────────────────────────────────────────────────┘

8. การจัดการสิทธิ์โดย org_admin (Admin Delegation)
    8.1 ขอบเขตการจัดการ
        org_admin สามารถ:
            - กำหนดบทบาทให้ผู้ใช้งานภายในหน่วยงานหลักและหน่วยงานย่อยที่ตนดูแล
            - สร้าง/แก้ไข/ลบข้อมูลโครงการภายในหน่วยงาน
            - กำหนดสิทธิ์ project_lead, staff ให้ผู้ใช้งานในหน่วยงาน

        org_admin ไม่สามารถ:
            - กำหนดบทบาทระดับสูงกว่าตนเอง (super_admin, univ_executive, campus_executive เป็นต้น)
            - แก้ไขข้อมูลนอกหน่วยงานที่สังกัด
            - ลบหรือแก้ไขโครงสร้างหน่วยงาน   
    8.2 Role Assignment Constraints
        org_admin สามารถกำหนดบทบาทได้เฉพาะ:
            - faculty_executive  (เฉพาะภายใต้หน่วยงานที่ดูแล)
            - unit_head          (เฉพาะภายใต้หน่วยงานที่ดูแล)
            - org_admin          (เฉพาะภายใต้หน่วยงานย่อย)
            - project_lead       (เฉพาะโครงการในหน่วยงาน)
            - staff              (เฉพาะภายใต้หน่วยงานที่ดูแล)

11. ข้อพิจารณาด้านเทคนิค (Technical Considerations)
    11.1 Authentication & Authorization
        - Authentication: ใช้ระบบ SSO ของมหาวิทยาลัย (หากมี) หรือ Email/Password พร้อม MFA
        - Authorization: ใช้ RBAC (Role-Based Access Control) ร่วมกับ ABAC (Attribute-Based Access Control) โดย Attribute คือ org_id + campus_id
        - Token: JWT ที่มี claim ระบุ active_affiliation_id เพื่อกำหนด Context ปัจจุบัน
    11.2 Data Privacy & Security
        - ต้องมีการเข้ารหัสข้อมูลที่สำคัญ (encryption at rest & in transit)
        - Audit trail สำหรับการเปลี่ยนแปลงข้อมูลสำคัญ
        - จำกัดจำนวนครั้งที่สามารถสลับบทบาทได้ในช่วงเวลาหนึ่ง (rate limiting)
        - บันทึกการ login/logout และการเปลี่ยน affiliation_id

    11.3 Performance & Scalability
        - ต้องรองรับการโหลดข้อมูลตาม affiliation_id ได้อย่างรวดเร็ว
        - ใช้ caching สำหรับข้อมูล frequently accessed (เช่น รายชื่อผู้ใช้, บทบาท)
        - แบ่งตารางตาม affiliation_id หรือใช้ partitioning สำหรับข้อมูลจำนวนมาก

    11.4 User Experience
        - Interface ต้องแสดง affiliation ปัจจุบันชัดเจน
        - ให้ user เลื่อนย้อนกลับไปยัง affiliation ก่อนหน้าได้ง่าย
        - แจ้งเตือนเมื่อ affiliation เปลี่ยนแปลง
        - รองรับ dark mode และ accessibility standards

    11.5 API Design Concept
        GET  /api/projects                  → ดึงโครงการตาม scope ของ active affiliation
        GET  /api/projects/:id              → ดูรายละเอียดโครงการ (ตรวจสอบสิทธิ์)
        POST /api/projects                  → สร้างโครงการ (ต้องมีสิทธิ์ edit)
        PUT  /api/projects/:id              → แก้ไขโครงการ (ต้องมีสิทธิ์ edit)

        GET  /api/users/me/affiliations     → ดึงรายการสังกัดทั้งหมดของผู้ใช้
        POST /api/session/switch            → สลับ active affiliation

        GET  /api/admin/users               → จัดการผู้ใช้ (org_admin+)
        POST /api/admin/affiliations        → เพิ่มสังกัดให้ผู้ใช้ (org_admin+)

    11.6 Middleware Authorization Check
        function checkAccess(requiredPermission):
            1. ดึง active_affiliation จาก JWT
            2. ดึง role, org_id จาก user_affiliations
            3. คำนวณ accessible_orgs ตาม Scope Resolution Rules (Section 6.2)
            4. ตรวจสอบว่า target_resource อยู่ใน accessible_orgs หรือไม่
            5. ตรวจสอบว่า role มี requiredPermission หรือไม่
            6. IF pass → allow, ELSE → 403 Forbidden

    11.7 Audit Trail
        ทุกการกระทำที่สำคัญควรบันทึก Log:
        Field	Description
        action	CREATE / READ / UPDATE / DELETE
        actor_user_id	ผู้กระทำ
        actor_affiliation_id	สังกัดที่ใช้ ณ ขณะนั้น
        target_resource	ทรัพยากรที่ถูกกระทำ
        timestamp	เวลาที่เกิดเหตุการณ์
        ip_addressIP Address

12. สรุป (Summary)
    ระบบใหม่ได้รับการออกแบบเพื่อแก้ไขข้อจำกัดหลักของระบบเดิม โดยมีการเปลี่ยนแปลงสำคัญดังนี้:
        1.Multi-Affiliation: ผู้ใช้ 1 คนสังกัดได้หลายหน่วยงาน พร้อมบทบาทและตำแหน่งที่แตกต่างกันในแต่ละสังกัด
        2.Hierarchical Access: รองรับการเข้าถึงข้อมูล 3 ระดับ (มหาวิทยาลัย → วิทยาเขต → หน่วยงาน)
        3.Granular Roles: ขยายจาก 4 บทบาทเป็น 10 บทบาทที่ครอบคลุมผู้ใช้ทุกระดับ
        4.View/Edit Separation: แยกสิทธิ์ดูและแก้ไขอย่างชัดเจน โดยผู้บริหารเป็นแบบดูอย่างเดียว
        5.Decentralized Admin: org_admin สามารถจัดการสิทธิ์ภายในหน่วยงานของตนได้
        6.Session Context: มี Session Switcher เพื่อให้ผู้ใช้สลับบทบาท/สังกัดได้สะดวก
